Effective Objective-C 2.0 （五）内存管理

Posted on 2016-08-22 Edited on 2019-10-21 In Home , iOS | 阅读量: 次

第二十九条：理解引用计数

retain 增计数
release 减计数
autorelease 待稍后清理autorelease pool时，再减少计数

对象创建出来时，其保留计数至少为1

第三十条：以ARC简化引用计数

若方法名以下列词语开头，则返回的对象归调用者所有

  • alloc

  • new

  • copy

  • mutableCopy

在应用程序中，可用下列修饰符来改变局部变量与实例变量的语义

__strong 默认语义，保留这个值

__unsafe_unretained 不保留这个值，这么做可能不安全，因为等到再次使用变量时，其对象可能已经回收了

__weak 不保留这个值，但是变量可以安全使用，因为如果系统把这个对象回收了，那么变量也会自动清空

__autoreleasing 把对象_按引用传递_给方法时，使用这个特殊的修饰符，此值在方法返回时自动释放

比如，想令实例变量的语义与不使用 ARC 时相同，可以使用__weak或__unsafe_unretained修饰符

block 块会自动保留其所捕获的全部对象，而如果这其中有某个对象又保留了块本身，那么就可能导致循环引用，可以用__weak局部变量来打破这种循环引用

注意：CoreFoundation对象不归 ARC 管理，开发者必须适时调用CFRetain/CFRelease

第三十一条：在dealdealloc方法中只释放引用并解除监听

把原来配置过的观测行为都清除掉，如果使用NSNotificationCenter给此对象注册过某种通知，那么一般应该在这里注销

第三十二条：编写“异常安全代码”是留意内存泄露问题

1.捕获异常时，一定要注意try块内所创立的对象清理干净

2.在默认情况下，ARC不生成安全处理异常所需要的清理代码。开启编译器标志之后可以生成这种代码，不过会导致程序变大，并且降低运行效率。

Effective Objective-C 2.0 （四）协议与分类

Posted on 2016-08-22 Edited on 2019-10-21 In Home , iOS | 阅读量: 次

第二十三条：通过委托与数据源协议进行对象间的通信

1.讲解了委托的具体使用方法，当创建delegate属性的时候，要把属性的语义定义成weak而不是strong，否则可能会造成循环引用的问题（引用环）。

2.如果协议中有很多个方法，而你可能需要不停的判断是否有代理响应其代理方法，那么可以不用重复的书写

[code lang=”objc”] if([_delegate respondsToSelector:@selector(方法名)]){ [_delegate 方法名]; } [/code]

而是这样写：

[code lang=”objc”] struct data{ unsigned int didReceiceDataOne : 1; unsigned int didReceiceDataTwo : 1; unsigned int didReceiceDataThree : 1; } _delegate; if(_delegate.didReceiceDataOne){ //yes } if(_delegate.didReceiceDataTwo){ //yes } … [/code]

如果需要频繁的调用数据源方法那么则可以进行这种优化，可以提高程序效率。

第二十四条：将类的实现代码分散到便于管理的数个分类中

就是讲解分类的详细用法。

1.使用分类机制把类的实现代码划分成易于管理的小块。

2.将应该视为“私有”的方法归入到名为private的分类中，以隐藏其实现细节。

第二十五条：总是为第三方的类的分类名称加前缀

1.向第三方类中添加分类时，总应给其名称加上你的专用前缀。

2.向第三方类中添加分类时，总应给其方法名加上你的专用前缀。

第二十六条：不要在分类中声明属性

属性是封装数据的方式，尽管从技术上说分类中也可以声明属性，但是这种做法还是要尽量避免，因为除了“class-continuetion”其他分类都不能合成实行所需要的实例变量。

第二十七条：使用class-continuetion分类隐藏实现细节

1.class-continuetion是一个特殊的分类（就是在.m文件上面声明的，可以再其中声明属性和方法）。

2.如果.h中的属性是只读类型，但是对内又想让自己使用这个属性那么就可以在class-continuetion中重新声明此属性，只是把只读属性换成读写。

3.把私有方法的原型声明放在class-continuetion中。

4.若想使类所遵守的协议不为人所知，则可于class-continuetion中声明。

第二十八条：通过协议提供匿名对象

@property （nonatomic ， weak）id delegate;

该属性的类型是id ，所以实际上任何类的对象都能充当这一属性，即便该类不继承自NSObject亦可以，只要遵守EOCDelegate协议就行。对于具备此属性的类来说，delegate就是匿名的。

对Warning: Attempt to present on whose view is not in the window hierarchy!的解决方案

Posted on 2016-07-28 Edited on 2019-10-21 In Home , iOS | 阅读量: 次

当我想从一个VC跳转到另一个VC的时候，一般会用

- (void)presentViewController:(UIViewController *)viewControllerToPresent animated: (BOOL)flag completion:(void (^)(void))completion;当然也可以用导航push。

可是昨天就遇到了题目中的warning，在stackoverflow找到了答案：点击打开链接

大意就是页面跳转必须在viewDidLoad和viewDidAppear之后才能进行。解决的办法就是确保页面跳转要在view load完毕之后进行。大神说可以通过设置延时来实现，通过实验我觉得这个不好控制，到底需要延时多少呢》

后来换了一个方案，在viewDidLoad里用

[selfperformSelectorOnMainThread:@selector(login)withObject:nilwaitUntilDone:NO];

把页面跳转的代码写进函数里，然后将 waitUntilDone 设为NO，就是viewDidLoad直接返回不用等login执行。这样就可以确保login里的页面跳转是在viewDidLoad之后执行。

另外，我还遇到下面这个情况：A页面跳转到B，在B里又封装了一个页面跳转（至C）。

也就是在前面的login函数里，从A跳转至B时会传入token、secret等用来获取用户授权的数据，然后在B里再调各大公司封装好的授权页面（如百度网盘、新浪微博）。

一开始我的login实现如下：

-(void)login

{

loginViewController *loginVC=\[\[loginViewControlleralloc\] init\];

loginVC.delegate=self;

\[self  presentViewController:loginVC animated:NOcompletion:nil\];

\[loginVC   auth:BAIDU\_API\_KEY   secret:nil  disk:BAIDU\];//这个函数里有跳转至授权页面

}

这同样会出现上面的问题，解决的办法就是把最后一行的函数放在completion的block里。

-(void)login

{

loginViewController *loginVC=\[\[loginViewController alloc\] init\];

loginVC.delegate=self;

\[self  presentViewController:loginVC animated:NO completion:^{\[loginVC auth:BAIDU\_API\_KEY secret:nil disk:BAIDU\];}\];

}

因为：

The completion handler, if provided, will be invoked after the presented controllers viewDidAppear: callback is invoked.

Effective Objective-C 2.0 （三）接口与API设计

Posted on 2016-07-08 Edited on 2019-10-21 In Home , iOS | 阅读量: 次

第15条：用前缀避免命名空间冲突

1.选择与你的公司、程序或者二者界有关联的名称作为类名的前缀。（苹果公司保留所有两位前缀的使用权限）。

2.若自己开发的程序库中用到了“第三方库”，则应为其中的名称加上前缀。

第16条：提供“全能初始化方法”

1.在类方法中提供一个全能初始化方法，并与文档中指明，其他的初始化方法都在调用 全能初始化方法。

2.若全能初始化方法与超类不同，则需要覆写超类中对应的方法。

3.如果超类的初始化方法不适用于子类，那么应该覆写这个超类方法，并在其中抛出异常。

第17条：实现description方法

1.多使用LLDB的po命令。

2.若想在调试的时候打印出更详细的对象描述信息，则应该实现debugDescription方法。

第18条：尽量使用不不可变对象

1.尽量创建不可变的对象。

2.若某属性仅可与对象内部修改，则在“class-continuation 分类”中将其由readyonly扩展为readwrite属性。

3.不要把可变的collection公开，而应该提供相关的方法，来修改对象中可变的collection。

第19条：使用清晰而协调的明明方法

1.方法名要言简意赅，读起来要像个句子才好（怎么感觉有点高端黑）。

2.遵守驼峰命名法，方法的第一个单词的首字母小写后面的单词首字母大写，类的所有单词的首字母大写。

3.方法名里面不要使用缩略后的类型名称（比如string 不要写成 str）

第20条：为私有方法加前缀

对于公共方法来说要修改其名称或者签名要三思，因为类的公共API不能随便修改，如改动，那么所有使用该公共API都需要更新代码，对于内部方法来说，修改是没有问题的。

为了区分私有方法和公共方法，作者建议在私有方法前面加 p_ + 方法名 的类型命名。

第21条：理解Objective-C的错误模型

1.只有当发生了可是整个应用崩溃的严重错误时，才使用异常。

2.在错误不那么严重的时候可以指派“委托方法”来处理错误，也可以把错误信息放在NSError对象里面，经由输出函数返回给调用者。

第22条：理解NSCopying协议

1.若想令自己所写的对象具有拷贝功能，则需要实现NSCopying协议。

2.如果自定义的对象分为可变版本和不可变版本，那么久就得同时实现NSCopying与NSMutableCopying协议

3.复制对象时需决定采用深拷贝还是浅拷贝，一般情况下都是尽量执行浅拷贝。

4.如果你写的对象需要深拷贝，那么可以考虑新增一个专门执行深拷贝的方法。

Effective Objective-C 2.0 （二）对象、消息、运行期

Posted on 2016-07-07 Edited on 2019-10-21 In Home , iOS | 阅读量: 次

继续记录Effective Objective-C 2.0，今天进行的第二章分为8条，总体来说很难，很底层，前面四条读后虽然也并没有完全吸收，但是确实受益匪浅。后面四条关于Objective-C的消息分发机制的原理和底层函数实现，先放放吧。

第6条：理解“属性”这一概念

这一条大致讲解了为什么用属性而不是用作用域来定义变量，存取方法的描述。不再赘述，重点记录一下内存管理语义的用法。

assign:只会针对“纯量类型”（CGFloat、NSInterger）

strong：表明了一种“拥有关系”，这种属性设置新值时，设置方法会先保留新值，并释放旧值，然后把新值设置上去。

weak：表明了一种“非拥有关系”，为这种属性设置新值的时候，设置方法既不会保留新值也不释放旧值，此特质同assign类似，然而在属性所指的对象被销毁时，属性值也会被清空。

copy：此特质与strong类似，然而设置方法并不会保留新值，而是将其拷贝当属性类型为NSString类型时，经常用此特质来保护其封装性。

atomic和nonatomic代表这线程的安全与否，在iOS开发中一般使用atomic，因为iPhone设备上使用线程安全（nonatomic）会严重影响性能，但是在Mac上使用就不会。

第7条：在对象内部尽量直接访问实例变量

1、self.firstName 和 _firstName区别：

直接访问变量不经过Objective-C的“方法派发”，编译器会直接去访问所在的内存，速度快。

2、直接访问变量不会调用其“设置方法（get 、 set方法）”，这就绕过了为相关 属性设置的“内存管理语义”。比如ARC下直接访问一个声明为copy的变量，那么并不会拷贝该属性，而是保留新值释放旧值。

3、直接调用并不会触发 “键值观察（KVO）” 通知。这样做可能会出现问题。

至于什么时候应该使用存取方法什么时候应该直接访问变量，有一个折中的方法：当需要设置属性值的时候使用“设置方法”，而获取属性值的时候直接访问变量。

使用这个做法的时候应该注意：

1、在初始化方法或者delloc方法中应该直接访问实例变量，因为子类会重写设置方法。

2、使用懒加载的之后，一定要使用“存取方法”来访问变量，否则变量将永远不会初始化。

第8条：理解“对象自等同性”这一概念

这条大概就是深入探讨了“==” “isEqual”“isEqualto”等方法的异同，以及比较不同对象的比较方法。

第9条：以“类族模式”隐藏实现细节

之前并不知道“类族”（类簇）的概念，书中举了一个例子，UIButtom的初始化方法中有一个方法：buttonWithType：（UIbutton *）type使用的时候只需要传入按钮的类型就可以返回不同类型的按钮，这些不同类型的按钮都是继承自UIbutton，而且理论上每个子类都可以实现个性化的操作。

Cocoa里的类族：系统框架中有许多类族，大部分collection都是类族。比如NSArray…

类族模式可以把实现细节隐藏在一套简单的公共接口后面。

第10条：在既有类中使用关联对象存放自定义数据（重要）

这一条介绍了关联对象的使用，对于关联对象的使用场景书中是这样描述的：“有时需要在对象中存放相关信息。这时我们通常会从对象所属的类中继承一个子类，然后改用这个子类对象。然而并不是所有的情况都能这么做，有时候类的实例可能是由于某种机制所创建的，而开发者无法令这种机制创建出自己所写的子类实例。Objective-C有一个强大的解决此问题，那就是关联对象”。

书中举例了一种常见的情况，就是当一个页面需要显示多个UIAlertView的时候，通常的做法就是给每个UIAlertView赋tag值，然后在其代理方法中根据tag判断不同的UIAlertView做出不同的操作。

但是作者说这种写法很乱、不太好。给出了一种使用关联对象解决的方案，具体代码有时间再放出来吧。

Effective Objective-C 2.0 (一)

Posted on 2016-07-06 Edited on 2019-10-21 In iOS | 阅读量: 次 
对这本书慕名已久，好多人也再想我推荐这本书，最近才决定买回来拜读一下，书不太厚，大概200页。纸上觉来终觉浅，对我而言还是得总结一下，效果会好一些，预计Effective Objective-C 会分七天总结，第一天的感觉就是这本书很适合进阶使用，值得一读。

第一条：了解Objective-C语言的起源

1. Objective-C为C语言的超集，为C添加了面向对象的特征，Objective-C使用动态绑定的消息结构，在运行时才会检查对象类型，接受一条消息之后，究竟应该执行和中代码，有运行环境而非编译器决定。

2.掌握C语言中的指针和内存对于学习Objective-C有很大帮助。

第二条：在类的头文件中尽量少引用其他头文件

1.当我们引入一个一个头文件的时候要想想，我是不是需要知道这个类中的所有细节？如果仅仅只是需要使用类本身那么用@class引入头文件就好（书中叫做：向前引用），虽然使用#import导入头文件并没有错，但是会有一系列的弊端：

（1）如果两个类的头文件相互导入，就会导致循环引用的问题，这就意味着两个类中会有一个无法正常编译。

（2）#import导入了.h的所有内容，这会增加编译时间。

当然，有些情况是必须要引入头文件的，如果你的类继承自某个基类，则必须要引入这个基类的头文件，同样的，如果你的类继承了某个协议，那么也是必须要要引入头文件并且不能使用向前引用的。（这里的协议指的是需要单独新建俩文件的协议类型）

第三条：多用字面量语法，使用一直等价的方法

1.应该使用字面量语法来创建字符串、数组、字典。与创建此类对象的常规方法相比，这没做更加简明扼要。

第四条：多用类型常量，少用#define预处理指令

1.如果说你只想定义一个常量在某个“编译单元”中使用，就少用#define 多用static const type name = ;这种类型。而这个命名规则是：若局限于某个“编译单元”内则前面加字母k，若常量在类之外可见，则通常以类名为前缀。

2.如果说需要声明一个外界可见常值变量，此常量通常放在“全局符号表”中，以便在编译单元之外使用，定义：

extern NSString *const name;

extern NSString *const name = @”value”;

第五条：使用枚举表示状态、选项、状态码

1、如果把传递给某个方法的选项表示为枚举类型，而多个选项又可以同时使用，那么就将个选项值定义为2的幂，以便通过按位或操作将其组合起来。

2、在处理枚举类型的switch语句中不要实现default分支，这样的话加入新枚举之后编译器就会提示开发者：switch语句并未处理所有枚举。

2.你的APP得支持IPV6.

我一开始感觉是XMPP的问题，因为XMPP的一系列的代理方法都没有执行，为了验证先配置了IPV6的环境（参考：https://developer.apple.com/library/mac/documentation/NetworkingInternetWeb/Conceptual/NetworkingOverview/UnderstandingandPreparingfortheIPv6Transition/UnderstandingandPreparingfortheIPv6Transition.html#//apple_ref/doc/uid/TP40010220-CH213-SW1）

果然在IPV4环境下功能正常连接上IPV6上之后就出现了不能登录的问题，具体表现就是第一条所述。

我尝试了一下的方法：

1.更新XMPP。因为当初项目并没有使用CocoaPods所以更新起来非常麻烦，无果。

2.把IPV4地址转换成IPV6。无果。

最终的解决办法是：

1.网络请求的宏，使用指向IP地址的域名，而不是IP地址。

2.因为XMPP里面是用Socket做的链接所以还要在XMPPStream文件的这个方法稍作修改： [code lang=”objc”] - (id)init { if ((self = [super init])) { // Common initialization [self commonInit]; // Initialize socket asyncSocket = [[GCDAsyncSocket alloc] initWithDelegate:self delegateQueue:xmppQueue]; [asyncSocket setIPv4PreferredOverIPv6:NO];} return self; } [/code]

最终测试完美解决！

NSMutableAttributedString的简单使用

Posted on 2016-03-29 Edited on 2019-10-21 In Home , iOS | 阅读量: 次

使用NSMutableAttributedString可以修改同一个Label中不同段文字的属性，之前有使用过NSMutableAttributedString来做图文混排表情键盘，但是那个时候理解不是特别深入，今天再次遇到，但是只是一些简单的用法。

[code lang=”objc”] //统一设置Label中的文字颜色和大小 -(void)settingLabelTextAttribute:(UILabel *)label LabelText:(NSString *)string{ NSMutableAttributedString *attributeString = [[NSMutableAttributedString alloc]initWithString:string]; if ([string hasPrefix:@”就”]) { [attributeString setAttributes:@{NSForegroundColorAttributeName : UIColorFromRGB(0x20bed9),NSFontAttributeName : [UIFont systemFontOfSize:12]} range:NSMakeRange(0, 4)]; //range:NSMakeRange的范围这块会比较容易出现问题，当更改的时候应该注意空格的问题 [attributeString setAttributes: @{NSForegroundColorAttributeName : UIColorFromRGB(0x646464),NSFontAttributeName : [UIFont systemFontOfSize:12]} range:NSMakeRange(5, string.length - 5)]; } label.attributedText = attributeString; } [/code]
